## Disposition Vocabulary

Every reference case carries exactly one disposition. The vocabulary is closed: no other values are permitted, and no in-scope case may carry more than one.

| Disposition | Applies to | Meaning | Required evidence on the row |
|-------------|-----------|---------|------------------------------|
| **Covered** | in-scope (core, CLI) | An existing Rust test asserts the same observable outcome (per the rubric below). | A `file::test_fn` pointer to the asserting Rust test. |
| **Gap** | in-scope (core, CLI) | No existing Rust test asserts the equivalent observable outcome; T4 must add one and flip the row to Covered. | None initially; becomes a `file::test_fn` pointer once T4 closes it. |
| **Excluded** | in-scope (core, CLI) | The case asserts a C#-specific implementation detail, mock call-orchestration, or legacy CLI surface with **no observable analogue** in the redesigned Rust port; it is intentionally not ported. | A one-line justification tied to a charter scope guardrail or redesigned-CLI rationale. |
| **Deferred** | AgentTools / MCP only | The case belongs to the AgentTools programmatic surface the charter defers beyond v1. **Reserved for AgentTools**; never applied to an in-scope case. | Listed at file granularity (per D4); no per-case pointer or justification required while deferred. |

**Closure rule**: each in-scope (core + CLI) case is dispositioned exactly one of {Covered, Gap, Excluded}. {Deferred} is reserved exclusively for the AgentTools area and is never a valid disposition for an in-scope case. ## Behavioral-Equivalence Rubric (REQ-004)

Equivalence between a reference C# test and a Rust test is judged **solely by observable outcomes**, never by mirroring C# class names, method names, internal types, or call structure. The Rust port deliberately embeds the Typst compiler in-process and exposes a redesigned CLI; insisting on structural mirroring would re-import the cruft the charter set out to drop.

### What counts as "the same observable outcome"

A Rust test is equivalent to a reference case when it asserts the same result across the observable surfaces the reference case constrains:

1. **Output value / text** — the produced Typst source, composed `.ndoc.typ` section text, or generated fat-file bytes match the behavior the reference asserts (for example "Markdown heading level 1 -> Typst `=`", or "compose produces a STATE section containing the document JSON"). Verified via `insta` snapshots for Typst/fat-file output or direct equality assertions for scalar results.
2. **Exit code** — for CLI behaviors, the process exit status (0 success / non-zero failure) matches what the reference CLI test asserts. Verified via `assert_cmd`.
3. **`--json` envelope shape** — for CLI behaviors emitting structured output, the `--json` envelope carries the same fields and structure the reference asserts (camelCase keys, arrays, nested document/collection shapes). Equivalence is about the **observable envelope**, not the producing API; field names follow the Rust `--json` contract, and only the asserted-upon structure must match.
4. **Emitted validation violations** — for validation behaviors, the set of `Violation`s reported (kind/code plus the offending key, node id, or source path) matches what the reference asserts. Equivalence is on the violation **set and the information each carries**, not on identical message strings.
5. **Error / failure mode** — when a reference case asserts a throw or error, the Rust equivalent asserts the corresponding failure (typed `Error` variant at the library seam, or non-zero exit + stderr at the CLI seam) for the same triggering condition. The failure **mode and trigger** must match; the exception type name need not.

A reference case may constrain more than one surface (for example a CLI test that asserts both an exit code and a `--json` envelope); the Rust equivalent must match every surface the reference case actually asserts upon.

### When equivalence does not require a Rust test

A reference case is dispositioned **Excluded — not contrived into a test — whenever it has no observable analogue**, specifically:

- **Mock call-orchestration / interaction tests** — reference cases (often NSubstitute-based) that assert a collaborator method was called with particular arguments, in a particular order, or a particular number of times, without asserting any user-observable output, exit code, envelope, or violation. The redesigned Rust CLI is exercised end-to-end via `assert_cmd` rather than by verifying internal call sequences, so these have no observable analogue.
- **C#-only structural / implementation-detail assertions** — cases asserting a C# type shape, DI registration, command-tree construction, or other internal that the Rust redesign does not expose as observable behavior.
- **Legacy CLI surface superseded by the redesign** — cases asserting a legacy command name, flag, or ergonomic the charter explicitly replaced; the redesigned behavior (if any) is covered under its own case, and the legacy-surface assertion is Excluded with that rationale.

Each Excluded row records a one-line justification naming the reason (mock-orchestration / C#-internal / superseded-surface) tied to the charter or redesign. **Excluded must never be used to hide an observable behavior gap**: if a case asserts any observable outcome that the Rust port should also guarantee, it is Covered or Gap, not Excluded. Reviewer sign-off (REQ-004) confirms no exclusion masks an in-scope observable gap.

## Parity-Map Row Schema
